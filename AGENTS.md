# AGENTS.md

Guidance for AI agents working in this repository.

## What this repo is

A Kubernetes [aggregated API server](https://kubernetes.io/docs/concepts/extend-kubernetes/api-extension/apiserver-aggregation/) that adds `SupportTicket` and `SupportMessage` resources to a Milo control plane. It is built on `k8s.io/apiserver` and follows standard Kubernetes API conventions.

## Key patterns

### API types (`pkg/apis/support/v1alpha1/`)

- Follow Kubernetes API conventions: `spec` for desired state, `status` for observed state.
- All new kinds need: type definition, `+k8s:deepcopy-gen` markers, registration in `register.go`, and code regeneration.
- After changing types, always run `task generate` to regenerate deepcopy and OpenAPI definitions.

### Scheme registration (`pkg/apis/support/install/install.go`)

- Types must be registered under both `v1alpha1` and `runtime.APIVersionInternal` (the internal hub version). This is required for server-side apply and other API machinery operations. See the existing pattern in `install.go`.

### Storage (`internal/registry/*/storage.go`)

- Each resource needs a `genericregistry.Store` with `TableConvertor` set — use `rest.NewDefaultTableConvertor(...)`. Omitting it causes a fatal crash on startup.
- `DefaultQualifiedResource` and `SingularQualifiedResource` must use the plural and singular resource names respectively.

### Strategy (`internal/registry/*/strategy.go`)

- Implement `PrepareForCreate`, `PrepareForUpdate`, `Validate`, `ValidateUpdate`.
- `NamespaceScoped()` returns `false` for all resources in this server (cluster-scoped).

### OpenAPI (`pkg/generated/openapi/`)

- `generated.openapi.go` is machine-generated by `openapi-gen` — do not edit.
- The stub file `zz_generated.openapi.go` was deleted; do not recreate it.
- Run `task generate` to regenerate after type changes.

## Development commands

```sh
task build           # build binary
task generate        # regenerate deepcopy + openapi
task test            # run unit tests
task fmt && task vet # format and vet
task dev:build       # build container image
task dev:load-image  # load image into kind cluster
task dev:deploy      # deploy to cluster (waits for rollout)
task dev:logs        # stream logs
task dev:status      # show APIService + pod state
```

## Demo environment

The `config/demo/` directory contains everything needed for a local Kind demo:
- `auth/` — Dex OIDC provider (in-memory, no database). Demo user: `demo@datum.net` / `password`.
- The portals use `AUTH_OIDC_ISSUER=http://dex.auth-system.svc.cluster.local:5556/dex` for server-side OIDC discovery, and the browser hits `https://localhost:30443/dex` via the Envoy HTTPRoute.

```sh
task demo:up     # full demo from scratch (~10 min)
task demo:down   # delete kind cluster
task demo:seed   # create demo tickets and messages
task demo:verify # health check
```

See `docs/demo.md` for prerequisites and the full walkthrough.

### Verifying demo changes

**All changes that affect demo behavior must be verified by exercising the browser UI, not just the API.** Use a headless Playwright browser (Chromium via `nix-shell -p chromium`) to:

1. Log in at `https://cloud.localhost:30443` as `demo@datum.net` / `password` (cloud portal) or `https://staff.localhost:30443` as a staff user (staff portal).
2. Navigate to the relevant feature and interact with it — click buttons, fill forms, observe the result.
3. Take screenshots at key steps and check for console errors.

**Do not** accept a passing `curl` or direct API call as evidence that the portal UI works. Routing bugs, auth flows, React rendering errors, and middleware issues are only caught by going through the browser.

To launch a headless browser test in this environment:

```js
// In a .mjs test script
import { chromium } from 'playwright';
const CHROMIUM_PATH = '/nix/store/<hash>-chromium-<ver>/bin/chromium'; // get via: nix-shell -p chromium --run "which chromium"
const browser = await chromium.launch({
  headless: true,
  executablePath: CHROMIUM_PATH,
  args: ['--ignore-certificate-errors', '--no-sandbox', '--disable-dev-shm-usage'],
});
const page = await (await browser.newContext({ ignoreHTTPSErrors: true })).newPage();
```

The nix Chromium path changes with version; get the current one with:
```sh
nix-shell -p chromium --run "which chromium"
```


## Things to avoid

- Do not omit `TableConvertor` from a `genericregistry.Store` — it panics at startup.
- Do not register types only under `v1alpha1` without also registering under the internal hub version — server-side apply will fail.
- Do not use `kubectl apply --server-side` in scripts unless you are certain the internal hub types are registered.
- Do not edit `pkg/generated/openapi/generated.openapi.go` by hand.
- The `zz_generated.openapi.go` stub returning an empty map must not exist — it shadows the real generated file.
